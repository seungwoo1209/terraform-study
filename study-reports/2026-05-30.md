#### terraform.tf: terraform 블록이 들어가며 provider과 terraform의 버전 정의

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.92"
    }
  }

  required_version = ">= 1.2"
}
```

- “terraform“ block → 이 디렉터리에서 쓰이는 terraform 자체에 대한 정의
    - provider, version 값이 들어감
        - provider: binary plugin 으로, cloud provider API 호출로 인프라 관리
        - Terraform 자체의 binary로부터 provider의 binary는 분리되어 있는 구조이며,
        이로 인해 Terraform에서는 provider를 바꿔가면서 API를 제공하는 여러 infrastructure vendor에 대해 인프라를 구성할 수 있게 됨.
        - source: `hostname/namespace/type` 형식으로 들어가며 hostname이 빠지면 기본값인 registry.terraform.io로 채워진다.
            - aws의 경우: `"hashicorp/aws"` . hostname 생략된 `namespace/type` 형식
            - 전체 이름은 `registry.terraform.io/hashicorp/aws`
        - version: normal operation 중 major version upgrades를 피하기 위해 사용해라.
            - `~> 5.92` : major version은 5로 고정, minor version은 92 이상이면 허용
- terraform 자체의 버전도 설정
    - `>= 1.2` : 1.2 버전 이상 이라면 OK
    - 현재 설치된 버전 확인 방법은 `terraform -version`

#### main.tf: 인프라 메인 구성

```hcl
provider "aws" {
  region = "us-west-2"
}

data "aws_ami" "ubuntu" {
  most_recent = true

  filter {
    name = "name"
    values = ["ubuntu/images/hvm-ssd-gp3/ubuntu-noble-24.04-amd64-server-*"]
  }

  owners = ["099720109477"] # Canonical
}

resource "aws_instance" "app_server" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t2.micro"

  tags = {
    Name = "learn-terraform"
  }
}

```

- **provider block과 “primary” infrastructure은 main.tf에 두는 것이 best practice**
    - 그리고 각자 연관된 인프라를 각자 파일에 두자.
- **provider block**에서 내 provider에 의해 관리되는 모든 리소스들의 공통 option을 설정
    - provider 값은 terraform.tf에 있는 required_providers 중 하나여야 함
    - 여러 provider block을 통해 여러 provider에 대한 지원 / 같은 provider의 다른 설정 e.g. 리전을 다르게 등이 가능.
- cloud provider의 api에 접근하기 위해선 물론 자격증명이 필요
    - 참고로 aws는 aws cli의 credential을 그대로 사용
- provider 블록은 전체 디렉터리(모듈) 안의 aws 리소스에게 공통적으로 적용
    - 하나만 다르게 하기 위해서는 alias라는 개념을 도입해야 한다고 함

#### Data 블록

- cloud provider에게서 리소스에 대한 정보를 가져오고 싶을 때

```hcl
data "aws_ami" "ubuntu" {
  most_recent = true

  filter {
    name = "name"
    values = ["ubuntu/images/hvm-ssd-gp3/ubuntu-noble-24.04-amd64-server-*"]
  }

  owners = ["099720109477"] # Canonical
}
```

- 위 코드는 filter를 만족하는 가장 latest AWS AMI를 가져오는 data block
- 이를 통해 리소스에 대한 ID를 **하드코딩할 필요 없이, dynamic한 설정이 가능**
- Data source는 여러 가지 속성을 가짐(id, arn 등)
- 이 예시에서: `data.aws_ami.ubuntu` 는 하나의 참조가능 주소가 된다

#### Resource 블록

- 내 인프라의 하나의 구성 요소에 대응

```hcl
resource "aws_instance" "app_server" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t2.micro"

  tags = {
    Name = "learn-terraform"
  }
}
```

- 위 코드는 EC2 인스턴스를 생성하기 위한 resource block
- Provider를 개발하는 사람들이 지원하는 resources들과 arguments를 정해 놓았음.
- 첫 번째 줄은 `resource "<resource type>" "<resource name>"` 형식으로 정의됨
    - `<resource type>` : 현재 예시에서는 `aws_instance`
    - resource type의 prefix: provider의 이름
    - 나머지: provider-defined (즉 terraform이 이름 붙인) 리소스 타입.
- resource type 과 resource name가합쳐져 **resource address**가 만들어짐.
    - 여기서의 resource address는 `aws_instance.app_server`
    - 이 resource address를 통해 코드에서 이 리소스를 참조 가능
- 예시 코드 설명
    - `ami`: 말 그대로 AMI. 아까 정의한 data soucre의 ID를 사용한 모습.
    - `instance_type`: t2-micro로 하드코딩
    - `tags`: AWS에서 Name 태그는 인스턴스 이름. 다른 태그 또한 지정 가능하고 AWS 인프라 운영 편의를 위해 베스트 프랙티스를 따라 지정하는 게 좋음.

#### terraform init

- current working directory의 구성에 맞는 provider를 다운로드함
- 생기는 파일:
    - aws provider 바이너리가 `.terraform` 안에 설치됨
    - `.terraform.lock.hcl` : workspace에 걸친 provider의 버전 정보가 저장을 통해 실행 간 consistency 보장

#### terraform validate

- resource name이나 지원하지 않는 argument 등에 대한 걸 잡아내는 용도
- 즉 한번쯤 하면 좋을 듯

#### 이제 인프라를 구성해 보자

- 우선 terraform은 내가 만들 인프라의 변화에 대한 execution plan을 만듦
- 이걸 보고 내가 ok → terraform이 provider를 사용하여 이 변화를 적용함
- terraform apply → execution plan을 출력한 후 yes → 적용

#### state 살펴보기

- 내가 인프라를 띄우면, terraform은 이 인프라의 모든 정보를 적은 `terraform.tfstate` 를 관리하게 된다.
- 이 state file을 통해 인프라를 lifecycle에 걸쳐 관리한다.
- `terraform state list`: resource와 data source를 보여준다.
    
    ```bash
    seungwoo@DESKTOP-KTK0141:~/terraform-study$ terraform state list
    data.aws_ami.ubuntu
    aws_instance.app_server
    ```
    
- `terraform show`: 모든 state 를 보여준다. (즉, terraform.tfstate의 내용)
- 내가 plan과 apply를 통해 인프라를 변경할 때, terraform은 다음을 조합하여 execution plan을 생성한다.
    - **state file에서의 last known state**
    - **현재 provider의 API로 조회한 상태**
    - **현재 terraform 코드**
- 주의: state file에는 비밀번호나 보안 키 등의 민감정보들을 모두 포함하고 있다.
    - 따라서 state file은 깃허브 등에 절대 커밋하면 안 된다.
    - state를 HCP Terraform으로 원격으로 팀원과 공유하며 관리 가능하다.
- 따라서 gitignore에서 state file은 꼭 넣어야 한다.
    - provider는 바이너리 파일이고 용량이 크므로 꼭 이것도 넣도록 하자.
    - 이외의 gitignore 템플릿 중에서 Terraform을 선택 가능하니, 이걸 쓰면 좋을 듯.
